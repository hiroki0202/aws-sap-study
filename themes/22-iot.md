## 座学

## Part 1: SAAからの差分 — IoTサービスの全体像

SAAではIoTにほとんど触れていません。SAPではIoTの基本アーキテクチャと主要サービスの使い分けが問われます。

**AWS IoTサービスの構成**:

| サービス | 役割 |
|---------|------|
| **IoT Core** | デバイス接続・認証・メッセージングのハブ |
| **IoT Device Management** | 多数デバイスの一括管理 |
| **IoT Device Defender** | セキュリティ監査・異常検知 |
| **IoT Analytics** | IoTデータの分析 |
| **IoT Events** | 複雑なイベントパターンの検知 |
| **IoT Greengrass** | エッジコンピューティング（デバイス上でLambda実行） |
| **IoT SiteWise** | 産業用IoT、工場のデータ収集 |
| **IoT TwinMaker** | デジタルツイン |
| **FreeRTOS** | マイクロコントローラ向けRTOS |

---

## Part 2: IoT Core — 中核サービス

**IoT Core**はAWSのIoT基盤の中核で、数百万〜数億台のデバイスを管理できます。

**主要コンポーネント**:

**Message Broker**: MQTT、HTTPS、WebSocketプロトコルでデバイスとAWSの間の通信を仲介

**Device Registry**: デバイスごとのメタデータ（ID、属性、認証情報）を管理

**Device Shadow**: デバイスの「あるべき状態」と「現在の状態」を AWS クラウド側に保持するJSONドキュメント。デバイスがオフラインの時でもアプリケーションから状態を参照・指示できる

**Rules Engine**: MQTTメッセージにSQL風のルールを適用して、他のAWSサービス（Lambda、DynamoDB、S3、Kinesis、SNS）に転送

**認証**:
- **X.509証明書**: 最も一般的、デバイスごとに証明書を発行
- **Cognito Identity**: ユーザー認証
- **IAM**: AWSサービス間連携
- **Custom Authorizer**: カスタム認証ロジック

---

## Part 3: Device Shadow — オフライン対応の鍵

**Device Shadow**は、デバイスとアプリケーション間の非同期状態管理を実現します。

**2つの状態**:
- **Desired**: 「こうあるべき」とアプリケーションが指示する状態
- **Reported**: デバイスが「今こうなっている」と報告する状態

```json
{
  "state": {
    "desired": { "temperature": 25 },
    "reported": { "temperature": 28 }
  }
}
```

温度センサーの目標温度を25度にアプリから設定。デバイスがオフラインだったら、オンラインになった時点でDesiredを検知して動作。結果をReportedとして報告。

**ユースケース**:
- 家電のリモート制御（エアコン、照明）
- 設定変更の配布（ファームウェアバージョン、設定値）
- 状態同期

---

## Part 4: Rules Engine — データルーティング

**IoT Rules Engine**は、MQTTメッセージに対してSQL風のクエリでフィルタリング・変換し、他のサービスに転送します。

```sql
SELECT temperature, deviceId, timestamp()
FROM 'sensors/+/temperature'
WHERE temperature > 30
```

このルールで「温度が30度を超えたセンサーメッセージ」を抽出し、下記アクションを実行。

**主要なアクション**:
- **Lambda**: カスタム処理
- **DynamoDB**: 最新状態の保存
- **Kinesis Data Streams / Firehose**: ストリーム処理・アーカイブ
- **S3**: オブジェクト保存
- **SNS**: 通知
- **Republish**: 別のMQTTトピックに再送信
- **IoT Events**: イベント検知に転送
- **IoT Analytics**: 分析パイプライン

---

## Part 5: Greengrass — エッジコンピューティング

**IoT Greengrass**は、デバイス上でLambda関数・コンテナ・MLモデルを実行できるエッジコンピューティング基盤です。

**ユースケース**:
- **ネットワーク切断時の自律動作**: 工場が一時的にインターネット切断でも処理継続
- **低レイテンシ処理**: センサーデータをデバイス上で即座に判定
- **帯域削減**: エッジで集計してからクラウドに送る
- **ローカルMLインファレンス**: エッジデバイスでMLモデル実行（SageMaker Neoと連携）

Greengrass Coreがハブデバイス（ゲートウェイ）で動作し、周囲のセンサーデバイスを統括します。

---

## Part 6: IoT Analytics と IoT Events

**IoT Analytics**: IoTデータの収集・変換・分析に特化。自動的なクレンジング、時系列データストア、SQLクエリ、QuickSight統合。

**IoT Events**: 複雑なイベントパターンの検知。「センサーAが30度超過」かつ「センサーBがOff」かつ「1時間以内」のような状態機械を定義。

## 練習問題

### 問題1

ある農業IoT企業では、10万台の土壌湿度センサーから毎分データを受信しています。以下の要件があります。

1. センサーはMQTTプロトコルで接続
2. X.509証明書でデバイス認証
3. データをリアルタイムでS3（長期保管）、DynamoDB（最新値）、異常値のみLambda（アラート）に振り分け

最適な構成はどれですか？

<details>
<summary>選択肢を見る</summary>

A. IoT Coreで全センサーを接続し、X.509証明書で認証。Rules Engineで SQL風のルールで3つのアクションに振り分け（Firehose → S3、DynamoDB アクション、Lambda アクション）

B. API Gateway + Lambdaで10万台のデバイスを処理する

C. EC2にMQTTブローカーを自作する

D. Kinesis Data Streamsに直接デバイスが書き込む

</details>

<details>
<summary>正解と解説を見る</summary>

**正解: A**

IoT Core + Rules Engineが正解です。

- **MQTT接続**: IoT Coreのメッセージブローカーが標準のMQTTプロトコルに対応
- **X.509認証**: IoT Coreのデバイスごとの証明書認証をサポート
- **Rules Engine**: SQL風のクエリで振り分け、1つのメッセージを複数のアクション（S3、DynamoDB、Lambda）に同時送信
- **スケール**: 10万台のデバイス、毎分のメッセージに対応

- B: API Gateway + Lambdaはデバイス接続に不向き（デバイス認証、永続接続MQTT非対応）
- C: 自作MQTTブローカーは運用負荷が高く、スケール・セキュリティ両面で劣る
- D: デバイスからKinesisに直接書き込むことはできるが、MQTT対応、デバイス認証、Device Shadowなどの機能がないため実用的でない

</details>

### 問題2

あるスマートホーム企業のエアコン制御アプリでは、ユーザーが外出先からエアコンの目標温度を設定します。しかし、エアコンは時々インターネット接続が不安定で、オフラインになることがあります。

オフライン時にアプリから温度設定の指示を出し、エアコンがオンラインになったタイミングで最新の指示を反映させたいです。

最適な構成はどれですか？

<details>
<summary>選択肢を見る</summary>

A. アプリがエアコンに直接TCP接続を維持し、設定を送信する

B. IoT Device Shadowを使う。アプリは Desired 状態に温度を書き込み、エアコンはオンライン時に Shadow の Desired を読んで適用、Reported で現状を報告する

C. SQS FIFOキューで設定を保持しエアコンがポーリングする

D. DynamoDBに設定を書き込み、エアコンが定期ポーリングで読み取る

</details>

<details>
<summary>正解と解説を見る</summary>

**正解: B**

IoT Device Shadowが正解です。

- **Desired vs Reported**: アプリは「あるべき状態」（25度）をShadowに書き込み、デバイスは「現状」（28度）を報告
- **オフライン対応**: デバイスがオフラインでもShadowには指示が残り、オンラインになった時に同期
- **差分同期**: Shadowは変更部分の通知を発行でき、効率的な同期
- **マネージド**: IoT Coreがオフライン時の状態管理を自動化

- A: 直接TCP接続はエアコンのオフライン時に通信できず、オンライン復帰時の同期メカニズムを自作する必要がある
- C: SQSでも実装可能ですが、最新状態のみ保持・状態同期の仕組みがなく、自作する必要がある
- D: DynamoDBポーリングは無駄なクエリが多く、リアルタイム性もない

</details>

### 問題3

ある工場では、製造ラインのセンサーデータをAWSに送っています。以下の問題があります。

1. 工場のインターネット回線が時々切断される（月に数回）
2. 切断中もセンサーデータの処理が必要（異常値検出で緊急停止など）
3. インターネット帯域が限られているため、エッジで集計してから送信したい

最適な構成はどれですか？

<details>
<summary>選択肢を見る</summary>

A. 全てのセンサーデータを直接 IoT Core に送る

B. 工場内にゲートウェイを設置し、IoT Greengrassを動かす。エッジでLambda関数を実行してリアルタイム異常検知・緊急停止、集計データのみクラウド送信。インターネット切断時も自律動作

C. オンプレのデータセンターでKafkaを動かす

D. 各センサーにLambdaを埋め込む

</details>

<details>
<summary>正解と解説を見る</summary>

**正解: B**

AWS IoT Greengrassが正解です。

- **エッジでLambda実行**: ゲートウェイ上でLambda関数を動かし、センサーデータの異常検知
- **オフライン動作**: インターネット切断中もローカルでLambda実行、緊急停止ロジック動作
- **エッジ集計**: センサーから届くデータを集計してからクラウドに送信、帯域削減
- **オンライン復帰時の同期**: 接続回復時にためていたデータをクラウドに送信
- **MLインファレンス**: SageMaker Neoと連携してMLモデルをエッジで実行

- A: 全データをクラウドに送る構成は、インターネット切断時に動作不能、帯域も消費過多
- C: Kafkaは自前運用の負荷が大きく、エッジ処理・クラウド同期を自作する必要がある
- D: センサーハードウェア自体にLambdaを埋め込むのは現実的でない（リソース制約）

</details>

### 問題4

ある設備監視企業では、各設備から「温度」「振動」「電流」の3種類のセンサーデータが流れてきます。以下の複雑な異常パターンを検知したいと考えています。

「温度が60度を超え、**かつ** 振動が閾値超過、**かつ** 電流が30A超過の状態が10分以上継続した場合」→ メンテナンス通知

単一センサーの閾値判定ではなく、複数センサーの時間的な相関を見る必要があります。

最適なサービスはどれですか？

<details>
<summary>選択肢を見る</summary>

A. IoT Coreの Rules Engineで SQL で判定

B. IoT Eventsで複雑なイベントパターン（状態機械）を定義する。「温度超過」「振動超過」「電流超過」の各イベントを組み合わせ、10分継続条件も記述可能

C. Lambdaで全センサーデータを処理し自作ロジックを実装

D. CloudWatch Metrics に全センサーデータを記録してアラームを設定

</details>

<details>
<summary>正解と解説を見る</summary>

**正解: B**

AWS IoT Eventsが正解です。

- **複雑なイベント検知**: 複数のセンサー値の組み合わせ、時間継続条件を状態機械で記述
- **Detectorモデル**: 状態（Normal / Warning / Alarm）の遷移をGUIで定義
- **アクション**: 異常検知時のSNS通知、Lambda呼び出し、他のIoTトピックへ発行
- **マルチデバイス**: 複数のIoTデバイスからのイベント相関

- A: Rules Engineは単一メッセージへの条件判定に向くが、複数ソース・時間継続の判定は複雑で不向き
- C: Lambda自作は実装負荷が高く、状態管理（過去10分のデータ保持）も自作する必要がある
- D: CloudWatchアラームは単一メトリクスの閾値判定で、複合条件には向かない

</details>

### 問題5

ある自動車部品メーカーの工場では、数千台の機械からSCADA・PLCプロトコル（OPC-UA、Modbus）でデータが出力されています。これらのデータをAWSに集約して分析したいですが、SCADA/PLCプロトコルはAWS標準サービスでは直接サポートされていません。

最適な構成はどれですか？

<details>
<summary>選択肢を見る</summary>

A. IoT Coreのカスタムコードでプロトコル変換する

B. AWS IoT SiteWiseを使う。産業機器（OPC-UA、Modbus）からのデータ収集に特化したサービスで、SiteWise ゲートウェイが OPC-UA / Modbus からデータを受信しAWSにアップロード。アセットモデル、計算式、ダッシュボードも提供

C. Lambda内で独自プロトコル変換を実装

D. 各機械にAWS SDKを組み込む

</details>

<details>
<summary>正解と解説を見る</summary>

**正解: B**

AWS IoT SiteWiseが正解です。

- **産業機器対応**: OPC-UA、Modbusなどの産業プロトコルをネイティブサポート
- **SiteWise Gateway**: オンプレミスのゲートウェイが機械と通信、AWSにアップロード
- **アセットモデル**: 機械の階層構造（工場→ライン→機械→センサー）をモデル化
- **計算式**: リアルタイム計算（平均、合計、カスタム式）
- **ダッシュボード**: SiteWise Monitorで可視化

- A: IoT Coreで独自プロトコル変換を実装すると運用負荷が高い
- C: Lambda自作は産業プロトコルの実装負荷が大きい
- D: 既存の工場機械にSDK組み込みは現実的でない

</details>

### 問題6

あるIoTサービス企業では、IoTデバイスの脆弱性を自動監査したいと考えています。以下を監視したいです。

1. デバイスの証明書・IAM ポリシーの過剰権限
2. 共有される証明書（1証明書で複数デバイスが認証）
3. 異常な接続パターン（同一デバイスが複数地域から接続）

最適なサービスはどれですか？

<details>
<summary>選択肢を見る</summary>

A. CloudTrailで手動で監査する

B. IoT Device Defenderを使う。Audit機能で設定の脆弱性を定期スキャン、Detect機能で異常な振る舞い（地理的に離れた接続、異常メッセージ量など）を機械学習で検出

C. Lambda自作でポーリング監視

D. GuardDutyを設定する

</details>

<details>
<summary>正解と解説を見る</summary>

**正解: B**

AWS IoT Device Defenderが正解です。

- **Audit**: 設定の脆弱性（過剰権限、共有証明書、CAの有効期限切れ）を定期チェック
- **Detect**: 振る舞いの異常（接続パターン、メッセージ量、地理的異常）を機械学習で検知
- **アラート**: 異常検知時にSNS通知、自動対応可能
- **IoT専用**: デバイス固有の脅威に特化

- A: CloudTrailの手動監査は運用負荷が高く、機械学習による異常検知もない
- C: Lambda自作は開発コストが大きく、機械学習による検知も難しい
- D: GuardDutyは汎用のセキュリティ検知でIoTデバイス固有の監査は含まない

</details>
