## Part 1: なぜOrganizationsが必要か

まず、「なぜ複数のAWSアカウントを使うのか」という問いから始めましょう。

多くの現場では、開発環境・ステージング・本番環境を1つのAWSアカウントで運用しているケースがあります。しかし、これには根本的な問題があります。開発者が誤って本番のS3バケットを削除するかもしれない。ステージング環境のコストが本番と混在して、どちらでいくら使っているかわからない。セキュリティインシデントが起きたとき、影響範囲が全環境に波及する。こうした問題を解決する第一歩が、**アカウント分離**という設計思想です。

アカウントを分けることで3つの分離が実現します。**セキュリティ分離**——あるアカウントが侵害されても、他のアカウントのリソースには原則アクセスできません。**請求分離**——環境ごと・チームごとのコストが明確になります。**環境分離**——本番の設定変更が開発環境に影響しないよう、完全に独立したリソース空間を持てます。

しかし、アカウントが増えれば増えるほど、管理が分散するという新たな問題が生まれます。「あのアカウントのセキュリティ設定はどうなっているんだっけ?」「ルートユーザーのMFAは全アカウントで有効になっているか?」「全体でいくら使っているんだろう?」——これを人力でアカウントをまたいで管理しようとすると、すぐに破綻します。

**AWS Organizations**はこの問題を解決するために生まれました。複数のAWSアカウントを1つの組織として統合管理し、一括請求（コンソリデーテッドビリング）によるコスト可視化、組織全体へのポリシー適用、そしてアカウントの一元管理を可能にします。Organizationsは「アカウントの集合体を管理する仕組み」であり、後述するSCPやOUはその上に成り立つ機能です。

## Part 2: SCPが生まれた背景

Organizationsが組織を束ねる仕組みを作ったとして、次の問題が浮かびます。「各アカウントのセキュリティポリシーを統一したい。でも、どうやって?」

IAMポリシーを使えばいいのでは、と思うかもしれません。しかし、IAMには致命的な限界があります。**IAMポリシーはルートユーザーには効きません**。ルートユーザーはAWSアカウントの絶対的な管理者であり、IAMの制約を受けずにすべての操作が可能です。つまり、開発アカウントの担当者がルートユーザーでログインしてしまえば、どんなIAMポリシーも無意味になります。

さらに、IAMポリシーは「各アカウント内」で個別に設定するものです。アカウントが10個あれば、10個のアカウントに同じポリシーを設定しなければならない。そしてその設定が本当に全部のアカウントで正しく適用されているか、継続的に確認しなければならない。これは現実的ではありません。

この課題に対して生まれたのが**SCP（Service Control Policy）**です。SCPは「組織の外側から、アカウント全体に対して適用される制約」です。ルートユーザーを含む、そのアカウント内のすべてのプリンシパル（ユーザー・ロール）に対して効きます。比喩的に言えば、**SCPは建物の入館ルール**です。いくら社員証（IAMポリシー）を持っていても、建物の入館ルールで「18時以降は入れない」と決まっていれば、誰も入れない。ルートユーザーであっても、その建物のルールには従わなければなりません。

SCPは「**権限の上限（ceiling）**を定義する仕組み」と理解してください。「このアカウントでは最大でもこれだけの権限しか使えない」という天井を設けるものです。その天井の中で、IAMポリシーによって各ユーザーに個別の権限を付与するイメージです。

## Part 3: SCPの動作原理

SCPの仕組みを正確に理解するために、評価ロジックを押さえましょう。

実際に使える権限は、**SCP と IAMポリシーの積集合**（AND）で決まります。SCPが「S3のPutObjectは許可」、IAMポリシーが「EC2の操作全般を許可」という場合、実際に使える権限は「S3のPutObject」のみです。SCPで許可されていない権限はIAMがどれだけ許可していても使えませんし、逆にIAMで許可されていない権限はSCPがどれだけ広くても使えません。両方で許可されているものだけが実際に利用可能です。

SCPの適用は階層構造で行われます。**ルートOU → 中間OU → アカウント**という継承ツリーがあり、上位のSCPは下位すべてに継承されます。ルートOUにアタッチされたSCPは、組織内のすべてのアカウントに影響します。あるOUにアタッチされたSCPは、そのOU配下のすべてのアカウントとサブOUに影響します。

ここで重要な初期状態があります。Organizationsを作成すると、デフォルトで**FullAWSAccess**というSCPがすべてのアカウントにアタッチされています。これはすべての操作を許可するSCPです。つまり、「何も制限しない」という状態が初期値です。

SCPの運用には2つのモードがあります。**Denyリストモード**はFullAWSAccessを残したまま、禁止したい操作だけをDenyで追加していく方法です。新しいサービスが追加されても自動的に許可されるため、運用が楽です。一方、**Allowリストモード**はFullAWSAccessを削除し、許可したい操作だけをExplicit Allowで明示的に書く方法です。セキュリティは高まりますが、新しいサービスを使うたびにSCPの更新が必要になります。SAPの試験では、よりセキュアな要件の場合にAllowリストモードが問われるケースがあります。

重要な例外が2つあります。まず、**ルートユーザーもSCPに縛られます**。これがSCPの最大の特徴であり、IAMと異なる点です。次に、**管理アカウント（マスターアカウント）にはSCPが適用されません**。これは意図的な設計です。もし管理アカウントを誤ったSCPで縛ると、組織全体の管理ができなくなるためです。この例外はSAPでよく出題されます。「なぜ管理アカウントでSCPが効かないのか」ではなく、「この要件を満たすためにSCPは適切か」という問い方で問われることが多いです。

## Part 4: OU設計パターン

SCPが使えるとわかったとして、「どのアカウントに、どのSCPをアタッチするか」という問題が残ります。アカウントが30個あって、それぞれに個別のSCPを管理するのは現実的ではありません。ここで**OU（Organizational Unit）**の出番です。

OUはアカウントをグループ化する仕組みで、OUにSCPをアタッチすることで、そのOU配下のすべてのアカウントに一括でSCPを適用できます。つまり、OU設計の本質は「**同じSCPを適用すべきアカウントをまとめる**」ことです。

最も基本的なOU設計は**環境別**です。`Dev OU` / `Staging OU` / `Prod OU` という3層構造が典型例です。Prod OUには厳しいSCP（リージョン制限、インスタンスタイプ制限など）をアタッチし、Dev OUにはより緩やかな制限を設ける。この分け方は直感的でわかりやすく、多くの現場で採用されています。

次に、**セキュリティ集約アカウント**のパターンです。AWS CloudTrail・AWS Config・Amazon Security Hub・Amazon GuardDutyのログや設定を、1つの専用アカウントに集約します。このアカウントはセキュリティチームだけがアクセスし、SCPで他のアカウントからの操作を制限します。ログを専用アカウントに送り込む設計にすることで、侵害されたアカウントの担当者がログを改ざんできない、という抑止力にもなります。

また、**例外OU**という設計パターンも重要です。規制要件の厳しいワークロード（金融・医療）、実験的なPoCワーク、特殊な要件を持つサードパーティサービスなど、通常のSCPでは動かせないアカウントを隔離するためのOUです。ルートOUの制約を一部緩めたり、独自のSCPを適用したりします。

一方でやってはいけないアンチパターンもあります。「ワークロードごとにOUを作る」というアプローチです。アプリAのOU、アプリBのOU……と増やしていくと、OUが乱立してどのSCPがどこに効いているか追跡困難になります。OUは「**セキュリティポリシーの単位**」で設計するべきであり、アプリケーションの単位で設計するものではありません。

## Part 5: SCP vs IAM vs Permissions Boundary

ここで混同しやすい3つの権限制御の仕組みを整理します。試験でも実務でも、この3つの違いを正確に理解しているかが問われます。

**SCP**は「**アカウント・OU単位**」で効く、**組織レベルの権限の上限**です。そのアカウント内のルートユーザーを含む全プリンシパルに対して影響します。「このアカウントではEC2しか使えない」という制約を組織管理者が一方的に設定できる点が特徴です。アカウントの担当者にはSCPを外す権限がありません。

**IAMポリシー**は「**ユーザー・ロール単位**」で効く、**アカウント内の権限付与**の仕組みです。ルートユーザーには効きません。アカウント管理者がユーザーやロールに対して「何をできるか」を定義します。SCPの天井の内側で、個別の権限を細かく設定するための仕組みです。

**Permissions Boundary**はSCPと似て「権限の上限を設ける」仕組みですが、適用範囲が異なります。SCPはアカウント・OU単位で組織全体に効くのに対し、Permissions BoundaryはIAMロール・ユーザー単位でアカウント内の委任制御に使います。詳細はテーマ2で扱います。

3つを一言でまとめると、SCPは「**組織が設ける天井**」、IAMは「**個人に渡す鍵**」、Permissions Boundaryは「**鍵の作り方に制限をかける鋳型**」です。

## Part 6: 頻出ユースケース

最後に、SAPでよく問われる具体的なユースケースを押さえます。

**リージョン制限**は最頻出のユースケースです。「東京リージョン以外ではリソースを作成してはいけない」という要件に対して、SCPのConditionに `aws:RequestedRegion` を使ってap-northeast-1以外へのリクエストをDenyします。これは規制要件（データが国外に出てはいけない）や、コスト管理（不意にバージニアでリソースを作るミスを防ぐ）の観点から重要です。

```json
{
  "Effect": "Deny",
  "Action": "*",
  "Resource": "*",
  "Condition": {
    "StringNotEquals": {
      "aws:RequestedRegion": "ap-northeast-1"
    }
  }
}
```

**インスタンスタイプ制限**も頻出です。GPUインスタンス（p3、p4系）やメモリ最適化の大型インスタンスは非常に高コストです。開発者が誤って使い続けることを防ぐため、SCPで禁止するケースがあります。

**S3の暗号化強制**は、`s3:PutObject`を呼ぶ際にSSE（Server-Side Encryption）の指定がなければDenyするSCPです。これによりアカウント内のすべてのS3書き込みで暗号化が強制されます。IAMポリシーで設定するより確実性が高く、ルートユーザーの操作にも適用されます。

```json
{
  "Effect": "Deny",
  "Action": "s3:PutObject",
  "Resource": "*",
  "Condition": {
    "StringNotEqualsIfExists": {
      "s3:x-amz-server-side-encryption": ["AES256", "aws:kms"]
    }
  }
}
```

**タグポリシー**はSCPとは別の仕組みですが、Organizationsの機能として重要です。タグの命名規則（`Environment`は`prod`か`dev`か`staging`のみ許可、など）を組織全体で標準化できます。コスト配分やリソース管理の前提となる基盤です。

最後に**AWS Control Tower**についてです。Organizations・SCP・AWS Config・IAM Identity Center（旧SSO）を組み合わせ、ランディングゾーン（マルチアカウント環境の基盤）を自動的にセットアップするサービスです。ガードレール（GuardrailsはSCPとConfigルールの組み合わせ）を適用し、**Account Factory**（アカウントを標準設定で自動プロビジョニングする機能）を提供します。「ゼロからマルチアカウント環境を構築せよ」という要件には、手動でOrganizationsを設定するよりControl Towerを選ぶほうが適切なケースが多く、SAPでも積極的に選択肢に出てきます。

## ハンズオン

### 前提条件
- AWSアカウントが1つ以上ある（フリーティア可）
- まだOrganizationsを使っていない場合は新規作成

### 手順

#### Step 1: Organizationsを作成する

1. AWSマネジメントコンソールにログイン
2. **AWS Organizations** サービスを開く
3. 「組織を作成」をクリック
4. 「すべての機能を有効にする」を選択して作成

#### Step 2: OUを作成する

1. Organizations コンソールで「Root」をクリック
2. 「アクション」→「組織単位を作成」
3. 以下の2つのOUを作成：
   - `Production`
   - `Development`

#### Step 3: テスト用SCPを作成する（リージョン制限）

1. 左メニューから「ポリシー」→「サービスコントロールポリシー」
2. 「ポリシーを作成」をクリック
3. 名前: `DenyNonTokyoRegion`
4. 以下のJSON:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyNonApprovedRegions",
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": ["ap-northeast-1"]
        },
        "ArnNotLike": {
          "aws:PrincipalARN": "arn:aws:iam::*:role/aws-service-role/*"
        }
      }
    }
  ]
}
```

5. 保存

#### Step 4: SCPをOUにアタッチする

1. 「組織単位」から `Development` をクリック
2. 「ポリシー」タブ → 「アタッチ」
3. 先ほど作成した `DenyNonTokyoRegion` を選択
4. アタッチ

#### Step 5: 動作確認

1. Developmentに所属するアカウント（または管理アカウント以外）にログイン
2. リージョンを `us-east-1` に切り替え
3. EC2 コンソールで何かアクションを試みる
4. **「権限がありません」エラーが出ればSCPが有効に動作している** ✅

#### Step 6: 継承の確認

1. Root にもう1つSCPを作成（例：`DenyDeleteCloudTrail`）
2. Root にアタッチ
3. Development配下のアカウントで、CloudTrail削除を試みる
4. **Root SCPもOU SCPも両方有効になっていることを確認**

#### クリーンアップ
- SCPをデタッチ → 削除
- OUを削除（中にアカウントがない状態にしてから）
- 不要であれば Organizations を無効化

## 練習問題

### 問題1

ある精密機械製造グループは、国内外に25のAWSアカウントをAWS Organizationsで一元管理しています。各拠点の開発チームは独自のOUに配置されており、CI/CDパイプラインを通じてAWS Lambda関数を定期的にデプロイしながら製造ラインの自動化システムを運用しています。

先週、クラウド基盤チームがセキュリティ強化施策の一環として、新しいService Control Policy（SCP）を組織全体に展開しました。翌営業日、複数の拠点の開発担当者から「Lambda関数のデプロイが突然できなくなった」という報告が相次ぎました。エラーメッセージにはアクセス拒否を示すコードが含まれていましたが、影響を受けたアカウントの数が多く、パターンが見えない状況です。

調査チームは各アカウントのIAMロールを確認したところ、Lambda関数のデプロイに必要な `lambda:CreateFunction` などのアクションが明示的に許可されており、IAMの設定には問題が見当たりませんでした。また、AWSのサービス障害情報も確認しましたが、Lambdaの障害は報告されていません。

この問題の原因として最も可能性が高いものはどれですか？

<details>
<summary>選択肢を見る</summary>

A. Lambda関数のメモリ上限を超えている

B. 各アカウントのIAMロールの信頼ポリシーが正しくない

C. 最近展開されたSCPでLambdaの操作が制限されている

D. AWS Configルールが非準拠と判定してブロックしている
</details>

<details>
<summary>正解と解説を見る</summary>

**正解: C**

SCPはIAMポリシーの上位に位置する「許可の上限」であり、SCPで許可されていないアクションは、IAMでいくら許可していても実行できません。複数アカウントで同時に発生している点から、組織レベルの設定変更（SCP）が原因と推定できます。

- A: メモリ上限はデプロイ可否に直接影響しない
- B: IAMは問題ないと明示されている
- D: AWS Configは検知のみで操作をブロックしない
</details>

### 問題2

ある国内大手証券グループでは、AWS Organizationsを使って証券取引部門・バックオフィス部門のAWSアカウントを管理しています。現在のOU構成は「Trading OU（証券取引システム）」と「BackOffice OU（事務処理システム）」の2つに分かれており、それぞれ10以上のAWSアカウントを抱えています。

金融規制当局からの指導を受け、セキュリティチームは全部門に対して「東京リージョン（ap-northeast-1）以外でのAWSリソース作成を禁止する」というSCPを適用することにしました。このSCPはTradingとBackOffice両方に共通して適用が必要です。

さらに、Trading部門では高頻度取引システムの安定稼働を優先するため、「承認済みのEC2インスタンスタイプ以外は起動禁止」という追加制限を近い将来適用する計画が進んでいます。セキュリティ担当者は、共通SCPを2か所に重複して管理することを避けたいと考えており、将来の制限追加や変更が1か所の修正で完結するような設計を求めています。

SCPの重複管理を避け、将来の変更も一か所で済むようにするには、どのOU構成変更が最適ですか？

<details>
<summary>選択肢を見る</summary>

A. 両方のOUに同じSCPを個別にアタッチし、Trading側にはもう1つSCPを追加する

B. 共通制限用の中間OUを作成し、TradingとBackOfficeをその子OUとして配置する。共通SCPは中間OUに、Tradingの追加制限はTrading OUにアタッチする

C. Trading OUをBackOffice OUの子として配置し、共通SCPをBackOffice OUにアタッチする

D. 全てのSCPを組織のRootにアタッチし、条件キーで適用対象を分ける
</details>

<details>
<summary>正解と解説を見る</summary>

**正解: B**

SCPは親OUから子OUに継承されるため、共通制限を中間（親）OUに集約し、個別制限は各子OUに配置するのがベストプラクティスです。これにより：
- 共通SCPの変更は1か所で完了
- 各部門の独自制限も柔軟に追加可能
- OU間の分離性も維持

- A: 共通SCPが2か所に存在し、変更のたびに両方更新が必要
- C: 継承の方向が逆。BackOfficeの制限がTradingに漏れる
- D: 条件キーでの分岐は複雑で保守性が低い
</details>

### 問題3

あるヘルスケア企業は、病院・クリニック・研究機関向けに医療データ管理プラットフォームを提供しており、AWS Organizationsで50を超えるAWSアカウントを一元管理しています。扱うデータには患者の診断情報や処方履歴が含まれるため、医療情報保護法および自社のセキュリティポリシーにより、S3バケットに保存するすべてのオブジェクトはカスタマー管理KMSキー（CMK）による暗号化が義務付けられています。

しかし、複数のアカウントで繰り返し問題が発生していました。開発スピードを優先するエンジニアが暗号化オプションを指定せずに `s3:PutObject` を実行してしまい、平文のままオブジェクトが保存されるケースが後を絶ちません。都度AWS Configのアラートで検知しているものの、発見から修正までのタイムラグが生じており、コンプライアンス監査で指摘を受けるリスクが高まっています。

セキュリティアーキテクトは、事後検知ではなく「技術的に暗号化なしのアップロードを不可能にする」予防的な制御を導入することを決定しました。50以上のアカウントそれぞれに個別の設定を行う運用は現実的でないため、組織全体を一元的に制御できる仕組みが求められています。

この要件を満たす最適な方法はどれですか？

<details>
<summary>選択肢を見る</summary>

A. 各アカウントでAWS Configルールを展開し、暗号化されていないオブジェクトを検出して通知する

B. IAMポリシーで全ユーザーにs3:PutObject時のSSE-KMS指定を必須とするポリシーを配布する

C. SCPでs3:PutObjectリクエストにs3:x-amz-server-side-encryptionヘッダーがない場合を拒否する

D. Amazon Macieを有効化し、暗号化されていないオブジェクトをスキャン・削除する
</details>

<details>
<summary>正解と解説を見る</summary>

**正解: C**

SCPを使えば、組織単位で「暗号化指定のないPutObjectを拒否」するルールを強制できます。これにより：
- 全アカウントに一括適用可能
- エンジニアの操作ミスを技術的にブロック
- 個別アカウントへのポリシー配布不要

- A: Configは検知と通知のみ。ブロックはできない
- B: IAMポリシーの配布は各アカウントで必要。一元管理にならない
- D: Macieはデータの検出が目的。リクエスト自体のブロックは不可
</details>

### 問題4

ある国立研究機関では、各研究プロジェクトが必要なAWSリソースを迅速に構築できるよう、IT部門の担当者にIAMロールの作成権限を委任しています。中央のクラウドガバナンスチームは管理負担を軽減するため、各プロジェクトの担当者が自分たちで必要なロールを作成・管理できる体制を整えてきました。

しかし先月、あるプロジェクト担当者が作業効率を優先し、`AdministratorAccess` ポリシーをアタッチしたIAMロールを作成しました。このロールの認証情報が外部に漏洩し、第三者が全サービスへのフルアクセスを取得するというインシデントが発生しました。調査の結果、付与できる権限の範囲に上限が設定されていなかったことが根本原因と判明しました。

ガバナンスチームは再発防止策として、プロジェクト担当者が引き続きIAMロールを作成できる環境は維持しつつ、そのロールに付与できる権限に技術的な上限を設ける仕組みを導入することにしました。SCPによるアカウント全体への制限とは異なり、「ロールを作る人」に対して「作れるロールの最大権限」を縛る仕組みが必要です。

この要件に最も適した制御手段はどれですか？

<details>
<summary>選択肢を見る</summary>

A. SCPでiam:CreateRoleを条件付きで制限する

B. IAM権限境界（Permissions Boundary）を管理者に適用する

C. AWS Configで作成されたロールの権限を監視し、違反時に通知する

D. IAMポリシーから管理者のiam:CreateRole権限を削除する
</details>

<details>
<summary>正解と解説を見る</summary>

**正解: B**

IAM権限境界（Permissions Boundary）は、「IAMユーザーやロールが持てる最大権限」を制限する仕組みです。管理者にPermissions Boundaryを設定すれば：
- IAMロールの作成は可能（操作自体は許可）
- ただし、作成したロールが持てる権限は境界内に制限される
- 「AdministratorAccess」のような広範なポリシーを付けても、実効権限は境界に収まる

- A: SCPはアカウント全体の制限であり、「作成するロールの中身」まではAWS Organizations以外から制御できない
- C: 監視は事後対応であり、防止にはならない
- D: 作成権限を削除すると委任の目的を達成できない
</details>

- C: 通知は事後対応。無効化を防止できない

### 問題6

あるSaaSスタートアップ企業は、創業から3年で急成長を遂げ、エンジニアリング・データサイエンス・インフラなど複数の専門チームが独立したAWSアカウントで開発を進めています。アカウント数は30を超え、AWS Organizationsで一元管理する体制が整っています。

セキュリティ担当者は、30を超えるアカウントへの個別設定は非現実的であることを把握しており、将来さらにチームが増えても追加設定なしに制限が自動適用されるスケーラブルな設計を求めています。

この要件に最も適した構成はどれですか？

<details>

- B: 個別アカウントへのSCP適用は管理煩雑

### 問題7

あるITサービス企業では、AWS Organizationsで開発・ステージング・本番環境を管理しており、夜間の自動バックアップジョブがRDSスナップショットを定期的に作成しています。このバックアップは障害復旧（DR）の要となるプロセスであり、万一失敗した場合はRPO（目標復旧時点）の達成に影響します。

### 問題8

ある大手損害保険・生命保険グループでは、AWS Organizationsを使って全社のAWSアカウントを管理しています。現在のOU構成は「共通基盤 OU（ネットワーク・セキュリティ管理）」「損害保険事業 OU」「生命保険事業 OU」の3層で構成されており、各OUには複数のAWSアカウントが配置されています。

金融庁のガイドラインへの対応として、損害保険事業と生命保険事業の両方に共通して適用が必要なSCPが5本存在します。主なものとして「東京リージョン以外でのリソース作成禁止」「CloudTrail無効化禁止」「承認済み暗号化方式以外の禁止」などがあります。現状では、この5本のSCPを損害保険事業OUと生命保険事業OUの両方に個別にアタッチしているため、1つのポリシーを変更するたびに2つのOUを別々に更新しなければなりません。昨年のガイドライン改訂時には、変更箇所の1つを更新し忘れたまま数日間運用してしまい、内部監査で指摘される事態が発生しました。

クラウド統括部門は、共通SCPの管理を一元化して「1か所変更すれば全体に反映される」設計に移行したいと考えています。同時に、損害保険と生命保険は事業ドメインが異なるため、それぞれ独自のSCPや設定も維持できる柔軟性も必要です。また、共通基盤OUは保険事業向けのSCPとは異なる制限が必要なため、共通化の対象から除外する必要があります。

SCPの管理効率を上げつつ、各事業の独立性を維持するためのOU構成変更として最適なものはどれですか？

<details>
<summary>選択肢を見る</summary>

A. 「保険事業」という中間OUを作成し、「損害保険」「生命保険」OUをその子OUとして配置する。共通SCPは「保険事業」OUにアタッチする

B. 共通SCPをすべて組織のRootにアタッチし、「共通基盤」OUには別途例外用のSCPを追加する

C. 「損害保険」と「生命保険」のOUを統合し、1つのOUで運用する

D. 各アカウントに直接SCPをアタッチし、OUレベルでの管理をやめる
</details>

<details>
<summary>正解と解説を見る</summary>

**正解: A**

中間OUを挟むことで、共通SCPを1か所に集約でき、変更も1回で2事業に反映されます。各事業OUには固有の制限を追加することも可能で、独立性も維持できます。

- B: Rootに全部置くと共通基盤にも適用されてしまう。例外処理が複雑化
- C: 統合すると事業間の権限分離ができなくなる
- D: 個別アカウントへの直接適用はスケールしない
</details>
